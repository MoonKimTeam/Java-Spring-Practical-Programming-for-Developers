# 자동 테스트

인수 테스트는 시스템이 비즈니스 요구사항을 만족해서 소유권을 넘기기 전에 수행하는 테스트 단계를 뜻합니다.  
다시 말해 인수 테스트란 시스템을 인수인계하기 전에 시스템이 비즈니스 요구사항과 일치하는지 마지막으로 검증하는 테스트 단계라는 의미입니다.  
인수 테스트 단계에 수행할 테스트는 일반적으로 테스트 수행 절차와 기댓값이 적힌 체크리스트로 관리됩니다.  

인수 테스트는 테스트 단계 중 가장 마지막에 수행하는 '단계'를 지칭하는 말입니다.   
그러니 인수 테스트 과정에는 수동 테스트가 있을 수도 있고 자동 테스트가 있을 수도 있습니다.

자동 테스트는 소프트웨어를 이용해 자동으로 시스템을 검증하는 테스트를 말합니다. 자동 테스트는 여러 유형이 있으며 구현 방법도 다양합니다.  
Postman 같은 도구를 사용하거나, 스크립트를 작성하여 API를 실행하고 결과 값을 비교하는 방식으로 자동 테스트를 만들 수 있습니다.

## Regression

소프트웨어 개발 분야에서 사용하는 Regression이란 '시스템에서 정상적으로 제공하던 기능이 어떤 배포 시점을 기준으로 제대로 동작하지 않게 되는 상황을 지칭합니다.  
그래서 Regression은 버그의 일종이며, Regression이라는 용어로 부르는 것보다 Regression bug(회귀 버그)라고 부르는 것이 좀 더 정확합니다.  
이러한 회귀 버그 문제를 해결하기 위해 테스트를 꼼꼼히 해야합니다.  테스트를 이용해 공통 코드가 초기 개발 의도를 지키고 있는지 능동적으로 감시하면 됩니다.

## 의도

수정된 공통 코드가 처음 만들어졌을 때의 개발 의도를 벗어나면 의존하는 클래스들의 기대를 배신할 수 있습니다.  
우리는 타인이 작성한 코드에서 의도를 어떻게 파악할 수 있을까요 ?  
테스트는 이 같은 상황에 유용합니다. 왜냐하면 테스트는 개발자의 의도를 드러내는 최고의 방법이기 때문입니다.

테스트를 이용하면 객체에 할당된 책임과 의도를 기술할 수 있습니다. 즉, 어떤 객체의 책임과 의도를 테스트로 작성하는 과정에서 객체에 할당된 책임을 다시 볼 수 있게 됩니다.  
우리가 작성하는 테스트는，객체가 책임을 제대로 수행하는가?’에만 관심이 있습니다. 그 결과, 테스트를 작성하면서 객체를 책임의 단위로 바라볼 수 있게 됩니다.
